# Product

## Register

product

## Users

Jarvis Desktop is for people who want the full Jarvis runtime in a native
window instead of a terminal: developers, researchers, technical operators, and
power users who need persistent sessions, tools, memory, skills, profiles,
voice, artifacts, and approval flows while they work.

## Product Purpose

Jarvis Desktop makes the same agent core available through a polished desktop
surface. Success means users can start, steer, inspect, and recover real Jarvis
agent work without losing access to sessions, settings, profiles, tools,
approvals, skills, memory, messaging, cron jobs, command surfaces, or classic
chat debugging.

## Brand Personality

Precise, capable, and calm. The J.A.R.V.I.S reskin should feel technical,
voice-first, luminous, and premium while still preserving the trustworthy
product ergonomics of Jarvis Desktop.

## Anti-references

Do not make the interface feel like a generic chatbot, a terminal-only wrapper,
a cheap neon sci-fi dashboard, or a Marvel copy. Avoid feature-hiding spectacle,
over-decorated controls, excessive teal, gratuitous motion, and any redesign
that replaces Jarvis-native backend/session/tool flows with parallel UI logic.

## Design Principles

1. Preserve the Jarvis engine: reuse existing stores, routes, hooks, events, and
   primitives instead of rebuilding transport or agent systems.
2. Keep power visible: approvals, tool activity, session state, profile/model
   state, and classic chat fallback must stay reachable and legible.
3. Make voice feel native: the orb should react to real mic and TTS signals
   while text workflows continue to work through the existing composer.
4. Prefer disciplined atmosphere: dark technical surfaces, electric-blue focus,
   thin hairlines, restrained motion, and readable labels over decorative noise.
5. Design for recovery: advanced routes and Classic Chat remain available when
   users need debugging, long transcript history, or lower-motion interaction.

## Accessibility & Inclusion

Respect the desktop app's existing accessibility patterns: keyboard-reachable
controls, visible focus states, i18n for user-facing strings, reduced-motion
alternatives for continuous effects, readable contrast, and non-visual status
text for orb state changes.
